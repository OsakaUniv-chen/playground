# PSSP 模型 —— 背景与现状记录 (2026-07-07)

## 这个项目是做什么的

PSSP = 预测**下一个说话人在声图（sound map）中的位置**。整个流程会先从麦克风阵列的
音频片段生成 "sound map"（通过波束成形得到的房间二维热力图），模型输入过去若干帧的
声图序列，预测未来的帧。真正重要的评估指标是**预测峰值位置的定位精度**（即最响/最可能
是说话人所在的位置），而不仅仅是逐像素的重建误差。

## 为什么要做这次记录

- 代码是独自写的，没有借助 AI，用了一段时间后有些陈旧、存在一些粗糙的地方。
- 已经采集了新数据：**更多的录制场次，setup 相同**（同样的房间/麦克风阵列/任务，
  不是新环境、也没有新的模态）。所以这次更像是"同分布下增加数据量/多样性"的情况。
- 项目负责人对当前的模型和代码**不是很有信心**，希望这次能真正提升下一说话人定位
  的预测性能，而不是简单地用新数据重跑一遍训练。信心缺口涉及四个方面（见下文），
  默认不假设任何一部分是正确的。
- 目标：提升声图中"下一说话人"预测的性能（即在留出集上降低峰值定位误差 /
  提高 SSIM-PSNR 等指标）。目前还没有定死具体的目标数值，可以进一步讨论。

## 仓库结构 (`train-pssp/`)

| 文件 | 作用 |
|---|---|
| `11/create_dataset.py` | 读取 ROS2 bag，做波束成形（依赖本文件夹之外的 `soundmap`/`acoular` 模块）生成声图和灰度相机图，按录制场次写出 `.npz`，路径写死在外接 SSD 上。属于个人/临时脚本风格——路径和 bag 名称列表都是硬编码的。 |
| `utils_all_load.py` | `NPZLoader`（加载并拼接 npz 文件为训练/测试张量）+ `CustomDataset`（按 clip_len/pred_len 切窗口）+ 共用工具函数：MSE/SSIM 损失、峰值距离指标、早停、数据增强（翻转+随机裁剪）、参数解析、可视化画图。 |
| `simvp.py` | SimVP 架构——卷积编解码器 + 时序"翻译器"（gSTA/TAU/IncepU 三种变体）。目前是唯一能完整跑通的模型路径。 |
| `train_all_load.py` | DDP 训练主循环，支持 `--method {vae, simvp, vdt}` 三种方法。训练日志用 Weights & Biases 记录，基于测试集损失做早停。 |
| `calc_metrics.py` | 独立的评估脚本：计算 MSE 和"峰值距离命中率"（预测峰值与真实峰值距离小于某像素阈值的比例），并与"直接重复最后一帧"这个朴素基线做对比。脚本里留了很多被注释掉的变体（k3/k7/k9 卷积核大小）和硬编码的数据集/bag 列表——更像是草稿脚本，不是干净的命令行入口。 |
| `results/configs/simvp_exp4.json` | 目前保存的一份实验配置。`results/` 目录下暂无模型权重（`.pt`）或画图结果。 |

## 读代码时发现的潜在问题/风险（尚未修复）

1. **`vae.py` 和 `vdt.py` 这两个源文件已经缺失**，`__pycache__` 里只剩下过时的
   `.pyc` 文件。现在如果用 `--method vae` 或 `--method vdt` 启动训练会在 import
   阶段直接报错。目前只有 `simvp` 方法能正常跑。
2. `__pycache__` 里还残留着一些孤立的缓存文件（`utils2`、`all_load_utils2`、
   `batch_load_utils`、`utils_all_load_64`），说明之前存在过其他版本的 loader/utils
   代码，但对应的 `.py` 源文件已经不在了——是之前迭代遗留下来的痕迹。
3. **训练损失目前只用了 MSE**，但实际目标（下一说话人定位）真正关心的是峰值距离
   ——训练目标和评估目标之间可能存在不一致，值得进一步排查。
4. `calc_metrics.py` 没有和训练用同一套配置机制打通（硬编码 bag 名称列表、硬编码
   外接 SSD 路径、大量被注释掉的代码路径）。
5. 多模态融合（`sm_ratio` 把声图和灰度相机图做加权混合）目前是像素层面的简单加权
   平均，不是可学习的融合方式。
6. `SimVP.forward` 里输出被硬编码只取第一个通道
   （`Y[:, :pred_len, :1, :, :]`），默认声图永远是第 0 通道。
7. DDP 相关设置里硬编码了 `CUDA_VISIBLE_DEVICES='0,1,2,3'`，world size 直接由
   `torch.cuda.device_count()` 决定。

## 需要重点排查的信心缺口（负责人指出，四项都在这次排查范围内）

- **模型架构选择**——SimVP（相对于 VAE+diffusion 路线，或其他架构）是否真的适合
  这个任务？
- **训练设置/超参数**——损失函数、学习率调度（`train_all_load.py` 里 scheduler
  当前是被注释掉的）、数据增强，训练过程是否真的收敛得好。
- **数据流程正确性**——npz 加载/索引/训练测试划分逻辑（`NPZLoader.get_index`/
  `load_data`，以及 `CustomDataset.__getitem__` 里的切窗口逻辑）是否正确，有没有
  细微的 bug（比如有效索引范围的 off-by-one、skip_frames 窗口导致的训练/测试
  数据泄漏）。
- **评估指标是否有效**——目前算出来的峰值距离/MSE 是否真的能代表"下一说话人
  预测得好不好"，以及 `calc_metrics.py` 里的基线对比是否公平/有意义。

## 这一轮不打算做的事

- 不是新环境/新任务——新数据基本是同分布的补充，所以这次不是主要针对
  泛化能力/domain shift 的问题。
- 不打算新增模态——除非排查后发现有必要，否则先用现有的模态。
