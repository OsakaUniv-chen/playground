# try-VLM — 音图能否被 VLM 理解？（P6 前置探针）

report3 的 P6 提案（`../../../word-wolf-exp-eval/report3/next-study/proposal.md` §2.4/§4.0）主打**角度 2：用预训练 VLM（不 finetune）
prompt 出行为决策**，其中一种把音图喂给 VLM 的方式是「**把 sound map 叠加/渲染成图像**」。
但一个前提尚未验证：**VLM 到底能不能看懂音图这张热力图？** 本目录就是实际试一试。

> 结论若为**能**：叠加图像表示可行，P6 角度 2 走得通。
> 结论若为**不能**：只能退回「文本方位角」表示，或需要更强模型 / 更好的音图预处理。

## 音图是什么（数据）

来自 `train-pssp/train-data/*.npz`（447 个 bag）：

| 数组 | 形状 | 含义 |
|---|---|---|
| `soundmap` | `(T,64,64) float32` | 麦阵波束成形能量图，画在**鱼眼相机像平面**上 |
| `gray_camimg` | `(T,64,64) uint8` | 与音图**空间对齐**的鱼眼灰度相机帧 |
| `tick_ts` | `(T,)` | 时间戳（2Hz） |

相机是**全方位鱼眼**：图像**中心＝传感器正下方**，方向向外辐射。我们把画面当**钟面**读：
上＝12 点、右＝3 点、下＝6 点、左＝9 点（顺时针）。

## 分阶段设计

- **阶段 1（本次，读图定位）**：只给音图热力图，问「最响的声源在几点方向」，与 GT 对比。
  纯粹测「**能不能读懂热力图**」这一基础能力。← 先跑通这个。
- **阶段 2（后续，跨模态）**：给鱼眼相机 + 音图，问「哪个方位的人在说话」，与 VAD 真值对比。
  更贴近 P6 实任务，但把「读图」和「视听推理」混在一起。阶段 1 通过后再做。

## 两个关键的数据现实（决定了实验设计）

1. **原始音图很少有单一干净峰**。用角度能量剖面（`common.source_info`）筛「单瓣、集中」的帧，
   **命中率仅约 5.5%**——94% 的帧是弥散或**双瓣**（两人对坐 → 上下都亮）。
   → 阶段 1 只用干净单瓣子集，先测**天花板**（连干净图都读不了就没戏）。这个低命中率本身是一个发现。
2. **声源在数据里严重堆在 12 点**（座位先验）。若直接用自然分布，模型「永远猜 12 点」就能骗高分。
   → 每个选中帧按**已知角度旋转**，使 12 个钟点方向均匀覆盖；**GT 从旋转后的图重新算**
   （＝VLM 看到的那张图），从而测的是**读图能力**而非先验。

## 音图表示（本次）

**纯 Jet 热力图**（无相机、无坐标轴/罗盘标注）。关键：**in-disc 对比度拉伸**——
约 25% 像素是鱼眼圆盘外的黑角（值=0），会把 min 拉到 0、令整盘发红；改用**圆盘内百分位
（p55–p99.5）**归一化后，声源瓣才凸显为红、安静区为蓝（见 `common.jet_rgb`）。
钟面约定只写在 prompt 里，不画在图上。

## 模型

**Qwen2.5-VL-3B-Instruct**（bf16，约 7GB）——能舒服跑在 4070 laptop（8GB），也是 P6 的
本地实机部署目标。换 `03_*.py --model` 可试 7B/4-bit 等更强变体（定天花板用）。

## 文件

```
common.py            数据加载 / 钟面几何 / 角度能量剖面 / 旋转 / jet_rgb 渲染
01_select.py         选干净单瓣帧 + 旋转均衡 → manifest.csv
02_render.py         渲染纯 jet 热力图 → images/  (+ qc_contact_sheet.png 人工核对)
03_probe_localize.py 阶段1：VLM 读图定位（4-bit，--bf16/--model/--manifest/--imgdir）
04_analyze.py        混淆矩阵 + 误差直方图（--csv 指定结果）
05_make_synth.py     合成对照：干净单高斯 blob → images_synth/ + manifest_synth.csv
manifest*.csv        选中帧 + GT（可留）
images*/ results/     生成物（gitignore）
```

## 跑法

```bash
source ~/.virtualenvs/wolf/bin/activate      # 必须在 wolf venv
python 01_select.py && python 02_render.py   # 选样 + 渲染（先看 qc_contact_sheet.png！）
python 03_probe_localize.py                  # 真实音图 60 帧（4-bit）
python 05_make_synth.py                      # 合成对照 60 帧
python 03_probe_localize.py --manifest manifest_synth.csv --imgdir images_synth \
       --out results/synth_results.csv       # 在合成对照上跑
python 04_analyze.py --csv results/localize_results.csv   # 混淆图
```

评分：clock-exact（随机 8.3%）、clock ±1（随机 25%）、quadrant（随机 25%）。

## 结果（2026-07-20，Qwen2.5-VL-3B-Instruct, 4-bit nf4）

| 数据集 | clock-exact | clock ±1 | quadrant | mean\|err\| |
|---|---|---|---|---|
| **真实音图** 60 帧 | 15.0% (随机 8.3) | 36.7% (随机 25) | 33.3% (随机 25) | 2.28 h |
| **合成 blob** 60 帧 | 11.7% (随机 8.3) | 30.0% (随机 25) | 25.0% (随机 25) | 2.60 h |

**结论：Qwen2.5-VL-3B（4-bit）读不了热力图的方位——退化成猜。**

- **输出退化**：60 帧里模型只用了 **3 个答案**（真实：`10`×35/`5`×18/`3`×7；合成：`5`×29/`3`×19/`10`×12），
  从不说 1/2/4/6/7/8/9/11/12。混淆矩阵是**竖条纹**——预测与 GT 行**完全无关**（见
  `results/*_analysis.png`）。略高于随机的几个点全靠这几个固定猜测与均衡 GT 的偶然重合。
- **不是音图的锅，是模型的锅**：关键在**合成对照**——即便是教科书级干净的单个高斯 blob
  （blob 明明在正上方，它也答"10 点"），3B 一样失败。所以瓶颈**不是** sound map 弥散/噪声，
  而是**3B 根本不具备"从图里读出色块方位"这一空间 grounding 能力**。
- 响应无实质推理，一律"The center ... is located at approximately N o'clock"，N 近乎常数。

## 对 P6 的意义 & 下一步

1. **本地 3B 这一档不够**——"把音图渲染成图像喂 VLM"这条路，用 Qwen2.5-VL-3B 走不通。
   直接给行为决策更不可能（连方位都读不出）。
2. **必须先定天花板**：用**更强模型**回答"到底有没有 VLM 能读懂音图"。
   - 本地：`--model Qwen/Qwen2.5-VL-7B-Instruct`（4-bit，8GB 勉强）；
   - 或**前沿 API**（Claude / GPT-4o / Gemini），一次性排除"模型太弱"。
   若强模型也读不了 → 图像叠加表示判死，退回**文本方位角**表示（proposal §4.0 的另一路）。
3. **量化 caveat**：本结果是 3B-**4bit**。理应不至于把干净 blob 都读崩，但不能排除量化有损；
   GPU 空时可 `--bf16` 复跑一遍排除该混淆（bf16 约 7.5GB，需 8GB 基本空闲）。
4. 若阶段1（读图定位）某个模型能过，再做**阶段2 跨模态**（相机+音图 → 谁在说）。当前不必做。

## 另一个独立发现（对 proposal 有用）

原始音图里**干净单瓣的帧只有约 5.5%**，其余弥散/双瓣（两人对坐上下都亮）。即便将来有能读图的
VLM，raw sound map 直接喂之前很可能仍需**预处理/去噪/角度化**。
